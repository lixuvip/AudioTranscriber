# VoiceScribe 性能档位与资源控制技术规格

VoiceScribe 的核心定位是在 macOS 上实现**完全本地、高并发且资源受控**的音频转写推理。由于不同的转写引擎（FunASR / VibeVoice MLX / Qwen3-ASR）底层实现的计算框架不同，本系统引入了动态性能分档控制，结合系统的物理内存和芯片规格推荐最适配置。

---

## 1. 核心控制参数解析

性能档位的划分主要受三个参数的影响，这些参数会被转换为子进程的环境变量或命令行参数，传递给后端的 Python 推理脚本：

### ① 线程数限制 (`threads` / `OMP_NUM_THREADS` / `MKL_NUM_THREADS`)
*   **物理意义**：决定了底层矩阵运算库（PyTorch C++ 算子端、CPU 版 BLAS 库）在执行单步计算时并发的 CPU 核心线程数上限。
*   **为何限制**：默认情况下，PyTorch 和 Numpy 会拉满系统上所有的虚拟核心，这在 18 核的 M5 Max 设备上会导致系统 CPU 瞬时占用率暴涨至 1800%，引起严重的设备发热和 UI 卡死。
*   **设定准则**：在高配机器上，线程数上限限制为**最多不超过物理性能核心的一半**（例如，在 12 个物理性能核的设备上，最大限制在 6 个核心线程内），确保 macOS 依然有充裕的核心来维持平滑的系统渲染和运行其他软件。

### ② 批处理时间窗 (`batch-size-s`)
*   **物理意义**：用于长音频的分段（Chunking）处理。将超长音频按设定秒数切分为多个子段，然后分批次输入 ASR 模型，最后将识别出的语音片段重新缝合。
*   **为何设定**：ASR 模型（特别是基于注意力机制的 Transformer 和 Qwen 语言模型）的内存消耗会随着单次输入音频的长度呈**指数级或二次方级**增长。如果不分段直接转写一个 2 小时的音频，会导致内存瞬间爆满，从而被 macOS 系统内核的 OOM (Out Of Memory) 强行杀死（退出码 137）。
*   **设定准则**：
    *   **低档 (60s)**：单次最大处理 60 秒音频片段。适合低内存（如 8GB/16GB）设备，确保内存峰值被控制在 4GB 以下。
    *   **中档 (90s)**：单次最大处理 90 秒音频。适合 16GB-32GB 设备，推理速度和内存控制均衡。
    *   **高档 (120s)**：单次最大处理 120 秒音频。适合 32GB 以上设备，推理速度最快，但在加载大型模型时需要更高的内存驻留。

### ③ 说话人片段缝合长度 (`merge-length-s`)
*   **物理意义**：用于说话人区分（Speaker Diarization）阶段。在 Pyannote 或 cam++ 识别出不同的说话人时间轴后，将同一说话人前后间隔小于该设定秒数的简短停顿和相邻的文字语句，自动物理合并为一个连续的段落。
*   **设定准则**：通常设定在 10s 至 15s 之间。较长（15s）的合并长度能生成段落感更好、不易碎片化的会议记录；较短（10s）的合并长度则适合高密度、多人高频交互的辩论或快速问答场景。

## 2. 三大引擎不同档位技术指标

系统在启动的预热检测阶段，会读取物理内存并测算可用的 GPU 显存，针对不同引擎应用以下约束矩阵：

| 引擎类型 | 性能档位 | CPU 线程数 | 批处理大小 (batch) | 缝合长度 (merge) | 建议最低物理内存 | 运行预期内存峰值 |
| :--- | :---: | :---: | :---: | :---: | :---: | :---: |
| **FunASR** | 低 | 1 - 2 | 60s | 10s | 8 GB | ~ 2.5 GB |
| | 中 | 2 - 4 | 90s | 12s | 12 GB | ~ 4.0 GB |
| | 高 | 2 - 6 | 120s | 15s | 24 GB | ~ 6.0 GB |
| **VibeVoice MLX** | 低 | 1 - 2 | 60s | 10s | 8 GB | ~ 4.0 GB |
| | 中 | 2 - 4 | 90s | 12s | 16 GB | ~ 6.0 GB |
| | 高 | 2 - 6 | 120s | 15s | 32 GB | ~ 10.0 GB |
| **Qwen3-ASR** | 低 | 1 - 2 | 60s | 10s | 8 GB | ~ 2.5 GB |
| | 中 | 2 - 4 | 90s | 12s | 16 GB | ~ 4.0 GB |
| | 高 | 2 - 8 | 120s | 15s | 32 GB | ~ 8.0 GB |

## 3. 设计决策与内存预检机制

为了确保软件的健壮性，系统在用户点击“开始转写”时，会在 Swift 前端执行一次**内存预检**：
1.  **动态测算**：通过系统 API 获取当前系统的物理总内存以及空闲的可用内存（`checkAvailableMemory`）。
2.  **降级策略 (Auto-Downgrade)**：
    *   例如：当用户设置的是“高档”，但预检发现当前系统可用内存不足时，系统不会直接抛出崩溃或强行运行，而是**在后台自动将本次任务降级为中档或低档**运行，同时通过通知面板和日志向用户报告：“系统内存紧张，已为您临时降档以防止闪退”。
    *   这一机制极大提升了应用在极端低内存或多任务并存情况下的生还率。
